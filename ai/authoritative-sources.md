# Authoritative Sources During Troubleshooting

During troubleshooting, different systems provide authoritative evidence for different questions.

AI-generated responses, screenshots, summaries, and second-hand interpretations should not replace authoritative technical data.

---

# Core Principle

> Always use the system closest to the event as the authoritative source.

---

# Authoritative Source Mapping

| Question | Authoritative Source |
|---|---|
| Did the request hit the WAF? | WAF logs |
| What IP did the hostname resolve to? | `dig`, `nslookup`, DNS server |
| Did traffic reach the load balancer? | LB logs |
| Did the application receive the request? | Application logs |
| Was TLS negotiated successfully? | LB / web server TLS logs |
| Did the request traverse a proxy? | Proxy logs |
| Was the response generated by the WAF? | WAF response/event logs |
| Did the server generate the error? | Web server / application logs |

---

# Common Anti-Patterns

Examples of non-authoritative evidence:

- “The application team said it’s the WAF.”
- “AI said WAFs can cause 403s.”
- Screenshots without timestamps
- Error summaries without raw responses
- Second-hand troubleshooting summaries

These may provide useful leads, but they should not be treated as proof.

---

# Recommended Troubleshooting Approach

1. Collect evidence
2. Identify authoritative sources
3. Validate request flow layer-by-layer
4. Use AI to assist analysis only after evidence is collected
